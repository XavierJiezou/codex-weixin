# 会话历史与 Web 续聊设计

## 目标

“会话”页面不仅管理元数据，还能打开任意受管会话，查看该 Codex thread 的历史用户消息和最终回复，并从 Web 页面继续同一 thread。微信与 Web 使用相同的会话记录，任一入口产生的新 turn 都能在另一个入口继续。

同时修复微信重复回复：同一微信 `message_id` 在一个账号内只能触发一次 Codex turn，即使轮询批次重复、同步游标回退或服务重启也不能再次执行。

## 数据来源

历史消息以 Codex 自身的 thread 记录为唯一来源。app-server 通过稳定的 `thread/read` 方法和 `includeTurns: true` 返回 turn 与 item，不在 codex-weixin 中再维护一份容易失真的聊天副本。页面只展示用户消息与 Codex 最终回复，隐藏推理、命令、补丁和桥接内部提示。没有 thread 的新会话显示空状态，第一次 Web 发送后写回新 thread id。

## 服务结构

所有微信账号共享 AccountManager 持有的一个 HybridCodexRunner。微信消息和 Web 续聊因此走同一 app-server 连接，也能统一使用 exec 回退。新增两个本机 API：读取会话消息，以及向指定会话发送文本。API 继续使用已有页面临时令牌保护写操作，并用 accountId + sessionId 做隔离校验。

运行状态增加有限长度的 `processedMessageIds`。账号监控在处理消息前原子登记 message id，已存在则直接跳过；列表只保留最近 1000 条，防止状态文件无限增长。该记录随账号状态保存，因此服务重启后仍有效。

## 界面

会话页先使用横跨整个工作区的独立微信账号筛选栏，再进入清晰的主从双栏：左侧只显示当前账号的紧凑历史会话列表和管理操作，右侧是消息时间线与固定在底部的输入区。用户消息使用绿色右对齐气泡，Codex 回复使用中性左对齐卡片，使用 Marked 解析 Markdown 并经 DOMPurify 清理后渲染。加载、空会话、无 thread、发送中和错误都有明确状态；窄屏下账号栏保持独立横向滚动，双栏改为上下布局，选择会话后聊天区域自然接在列表下方。

`@im.bot`、`@im.wechat` 和 Codex thread id 都属于服务路由所需的内部标识，不在普通管理界面展示。当前 iLink 扫码确认、消息和配置接口没有返回微信昵称、头像，也没有公开的联系人资料查询接口，因此界面采用稳定的本地序号和默认图标，不抓取或伪造个人资料。

## 验证

自动化测试覆盖持久去重、thread 历史解析、API 权限与续聊写回 thread id。使用本机 Codex app-server 读取真实历史并继续一轮；构建后检查桌面和手机宽度的会话页面，最后重启服务并确认现有账号和会话仍在。
