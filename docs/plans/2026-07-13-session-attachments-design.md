# Web 会话附件设计

## 目标

微信中由 Codex 发送的视频、图片和文件，也应出现在 Web 会话历史的对应回复下。Web 历史继续以 Codex thread 为唯一来源，不额外复制聊天记录或媒体文件。

## 数据流

读取 thread 历史时解析 `codex-weixin-actions` 动作块和本地 Markdown 文件链接。动作块的文字部分继续作为回复正文，`send` 动作转换为附件元数据：类型、文件名、大小、可用状态和在该消息中的序号。浏览器响应不暴露本机绝对路径。

## 文件访问

新增会话范围内的只读附件地址，地址包含账号、会话、消息和附件序号。服务每次请求都重新读取对应 thread 并确认该路径确实来自该条 Codex 回复；不提供接受任意路径的通用文件接口。视频支持 HTTP Range，便于浏览器加载元数据、播放和拖动进度。文件不存在时仍显示历史附件名称，但标记为已移动或删除。

## 界面

视频使用原生播放器，图片使用受限尺寸预览，普通文件使用紧凑文件行；全部提供下载按钮。附件和回复正文处于同一个消息气泡中，不创建独立会话消息。聊天消息 DOM 只在历史内容变化时更新，后台五秒状态刷新不会重建正在播放的视频。

## 验证

自动化测试覆盖动作解析、路径隐藏、会话隔离和视频 Range 响应。使用现有“邹学超 / 会话 1”中已发送的 MP4 验证播放器非空、可读取时长、下载地址有效，并检查桌面与手机布局。
