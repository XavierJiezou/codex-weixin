# 单实例与重复投递修复设计

## 现场原因

机器上同时存在当前服务和一个旧的 `codex-weixin --help` 进程。旧版本会在绑定 Web 端口前启动微信账号监控，因此即使随后端口绑定失败，监控和它自己的 Codex app-server 仍可能留在后台。同一微信消息会被两个进程分别处理，消息 ID 去重无法跨进程生效。

## 修复

继续保留现有的账号级同步游标和消息 ID 去重，同时增加状态目录级 `service.lock`。一个 `~/.codex-weixin` 同一时间只允许一个服务进程运行；即使使用不同 Web 端口，第二个进程也会在启动微信监控前退出。锁记录 PID，正常退出时删除；异常退出留下的锁在确认 PID 已不存在后自动替换。

启动入口显式处理 `-h` 和 `--help`，只输出帮助信息。其他未知参数直接报错，不再被忽略后误启动服务。现有“先绑定 HTTP，再启动账号监控”的顺序继续作为第二层保护。

## 验证

自动化测试覆盖帮助参数、未知参数、同状态目录的并发实例以及失效锁恢复。实际部署时清理旧幽灵进程，重启新版服务，并确认系统中只剩一个 `codex-weixin` 父进程及其一个按需启动的 Codex app-server。
