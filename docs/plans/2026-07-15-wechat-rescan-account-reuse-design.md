# 微信重新扫码账号复用设计

## 问题

微信 iLink 在同一微信账号重新扫码时可能返回新的 `ilink_bot_id`，但稳定的 `ilink_user_id` 不变。当前服务直接把 bot ID 作为本地账号主键，因此重新扫码会创建一个没有备注、授权和会话的新账号。

## 设计

- 扫码完成后先按 bot ID 精确匹配；没有命中时，再按稳定的 `userId` 查找本地账号。
- 只有唯一的 `userId` 匹配时才自动复用，避免存在历史歧义记录时误合并。
- 复用时保留旧本地 `accountId` 和备注，从而继续使用原账号的 runtime 状态、授权、会话与入站附件目录。
- 使用本次扫码返回的 token、服务地址和保存时间覆盖旧凭据，并将账号设为启用。
- 已运行账号必须停止旧监听并用新凭据重新启动；新扫码结果返回旧账号身份，Web 页面不会新增账号卡片。

## 兼容与迁移

- 首次扫码和缺少 `userId` 的旧协议响应仍按 bot ID 保存。
- 如果同一 `userId` 已有多个历史账号，通用扫码流程不自动猜测目标。
- 对已确认的新旧重复记录，可指定旧账号为目标执行一次迁移：保留旧状态，采用新凭据，再删除空的新记录。

## 验证

- 同一 `userId`、不同 bot ID 的再次扫码只保留一个账号。
- 旧备注、授权和会话状态保持不变。
- 运行中的旧监听会停止，并使用刷新后的账号重新启动。
- 不同 `userId` 不会被合并。
